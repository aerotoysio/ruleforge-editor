# RuleForge editor — roadmap & agent handoff

> Handoff for the next agent/developer. Latest milestone: the **AI-core pivot (Phases 1–2)** shipped. This file is the to-do for what's next + the context to not regress the design.

## What RuleForge is
**AI-authored, human-verified, engine-executed** business rules.
- **Editor** (this repo, Next.js 16): the authoring UI. AI drafts rules; humans review / correct / approve.
- **Engine** (`../ruleforge`, .NET): deterministic DAG evaluator. Runs the compiled rule in the request hot path. **The LLM is never in the request path** — it only authors offline.
- **Workspace** (`../ruleforge-sample-workspace`): the rules, node defs, reference tables, templates, assets.

## Architecture keystones (do not regress)
1. **LLM authors; engine executes.** Claude runs only at author/review time and emits a normal rule JSON, validated against the node schemas, then executed deterministically.
2. **Nodes are the AI's typed target vocabulary** (~20 generic engine categories — see `src/lib/rule/compile-to-engine.ts`). Domain specificity = reference *data*, not new node types. This is what keeps AI output auditable + runnable.
3. **Editability is layered:** read (explanations) → NL-correct → precise form-edit → raw JSON. Don't remove the manual editors; they're the correction layer.
4. **Rule persistence spreads** (`writeRule`/`readRule` in `src/lib/server/workspace.ts`), so new top-level `Rule` fields round-trip automatically — that's how `groups` and `aiMeta` were added.

## Done (current state)
- **Phase 1 — AI front door:** guided new-rule **wizard** (`src/app/rules/new/NewRuleClient.tsx`) → **`/api/ai/author`** (policy text/PDF → input+output schemas + graph + per-node explanations + narrative + citations + tests; validate-then-repair via `compileRuleForEngine`).
- **Phase 2 — review-first:** nodes show `aiMeta.nodeExplanations` (`src/components/flow/NodeView.tsx`); **Summary tab** (`src/components/flow/RuleSummaryTab.tsx`) with clickable policy citations.
- AI = **Claude** (Anthropic SDK), prompt-cached catalog. Settings has a Claude provider (key + model), defaults to Claude. Key resolves `settings.anthropicApiKey || process.env.ANTHROPIC_API_KEY`.
- Editor/nav overhaul: rules **category tree** + row menu (Edit/Test/Duplicate/Delete); node **grouping** boxes; **asset-from-library** picker; **calc** (NCalc) expression builder; unified node config dialog; `/test` page removed (testing lives on the rule row + in the editor).
- Engine: **multi-field Set mutator** (`../ruleforge` `MutatorConfig.Sets`).
- **Two new node types** (shipped this session):
  - **Parse string** (`node-parse-string`, engine category `textParse`): splits a packed string (e.g. an SSR `"SSR BIKE P1 F OVERSIZE"`) into named tokens via a friendly `{token}` pattern (**no regex** — literals between braces are the delimiters, last token takes the remainder) and maps them onto a saved **Asset**. Engine: `RuleRunner.ExecuteTextParse` + `ParseByPattern` + `Models/TextParseConfig.cs`. Editor: unified `TextParseEditor` (token chips + a live preview that mirrors the engine parser char-for-char + asset picker + token→field mapping).
  - **Product node = asset-only:** the Product editor is now just a saved-asset dropdown (no template / no free-literal / no inline value editing) for non-technical users; value tweaks happen in **downstream** Set/calc nodes so they're visible in the workflow.
  - Both use a shared **`asset` PortBinding kind** (`src/lib/types/node-def.ts` + `AssetPicker.tsx`) that **compile inlines to the asset's `values`** (`compile-to-engine.ts`) — the engine never sees an "asset" concept, just a base object. Adding a binding kind means updating every exhaustive `kind` switch (`NodeView.formatPortBinding`, `validate.ts`, `NodeDesigner.tsx`).
- **Filters → multi-condition (the "conditions-list" pattern)** — Number, Date and Text filters each now hold a **stack of conditions on one field**, combined by **ALL/ANY**, instead of one predicate per node ("flight number ≥100 AND ≤199 AND in {list}", "weekday AND after 06:00 AND before a date", "cabin in {reference column} AND not in {Y,M,B}" — all single nodes).
  - **Engine:** each `*FilterConfig` (String/Number/Date) gained `Conditions: IReadOnlyList<*FilterCompare>?` + `Match` ("all"/"any"); the evaluators loop the compares and combine (single-compare path unchanged → back-compatible). `RuleRunner.ExecuteFilter` enrichment shows `match:all/any` for multi.
  - **Editor:** `NumberConditionsEditor` / `DateConditionsEditor` / `TextConditionsEditor` (`src/components/bindings/`), special-cased in `NodeConfigDialog`'s `primary.map` by source-port **type** (number/integer, date) — Text is keyed on `def.id === "node-filter-string-in"` so it doesn't hijack the veneers. Conditions live in `bindings.extras.conditions` (+ `match`, + `timezone` for date, `caseSensitive` for text); `compile-to-engine` maps them via `compileNumber/Date/StringCondition`. Date uses native date/time inputs (calendar popups, **no new deps**) + day-of-week/month chips + a shared timezone; Text supports a typed list **or a reference-table column** (in/not_in = valid/invalid → pass/fail). Card summary shows "N conditions · all/any" (`NodeView.describeBindings` reads `extras.conditions`).
  - **To add multi-condition to another filter type:** add `Conditions`+`Match` to its config, loop in its evaluator, add a `*ConditionsEditor`, special-case it in `NodeConfigDialog`, and emit `conditions[]` in `compileFilterConfig`. Filter node-defs must NOT mark `operator` `required` (the conditions list supplies it, not the port).
  - **Redundant filters hidden** — `node-filter-cabin / pax-type / loyalty-tier / day-of-week / number-range` now carry `"hidden": true` (new **`NodeDef.hidden`** flag; `RightPalette` filters hidden nodes out of the Add-nodes rail). Kept on disk so existing rules still load + compile; replaced by: the generic **Filter — text** (reference-column) for cabin/pax/loyalty, the Date filter's day-of-week condition, and the **Filter — number** `between` condition for number-range. Palette is down to **4 filters**: text, number, date, markets. **Markets kept** — its hierarchical airport picker is genuinely special. (`node-filter-number-compare` was renamed "Filter — number".)
- **Join / Enrich node** (`node-join`, engine category `join`): correlates a left array with a right array by key (e.g. `passenger.id = flight.paxId`) and attaches the matches under a named field; modes `collect` (array of matches) / `first` (single). Engine: `RuleRunner.ExecuteJoin` + `Models/JoinConfig.cs` — an **empty `right` reads the upstream node's output**, so it composes after a loop/parse/collect. Editor: sentence-like `JoinEditor` (left/right array pickers + **schema-aware key dropdowns** built from each array's item fields + collect/first toggle + a live plain-English summary). Demos: `rules/join-demo.json` (pax × flights) and `rules/join-ssr-demo.json` (For-each → Parse → Collect → Join: loose SSR strings reunited with their passengers).
- **Filter-list node** (`node-filter-list`, engine category `filterList`): keeps the elements of an array whose chosen field meets a stack of conditions (ALL/ANY) — the missing array-filter primitive that composes with Join (filter the right list before, or the result after). Engine `RuleRunner.ExecuteFilterList` **reuses the typed filter evaluators per element** (each element treated as the request root) — zero evaluator duplication; `Models/FilterListConfig.cs`. Editor `FilterListEditor`: array picker + Text/Number/Date toggle + the reused `*ConditionsEditor` fed the array's *item* schema (so its field picker offers element fields). Plus an **"only keep matched" (inner-join) toggle** on the Join node (`JoinConfig.OnlyMatched`). Demo: `rules/filter-join-demo.json`.
- **Engine compute time, surfaced + visualized** — `Envelope.DurationMicros` records pure DAG-eval time in **microseconds on every run** (was Debug-gated + millisecond-only, so a sub-ms engine read as 0). New **Engine Playground** (`src/app/playground/page.tsx`): pick any rule, run it, and see engine compute (µs) vs the loopback call vs test-harness staging, with a 25× p50/p95 benchmark + a reset. Warm compute ≈ 0.1–0.3 ms.
- **SQLite workspace (`workspace.db`) — the editor + engine are two connected apps over one DB.** Engine: `SqliteRuleSource` + `SqliteReferenceSetSource` (`RULEFORGE_RULE_SOURCE=sqlite`, WAL; in `RuleForge.Core/Loader/`) read `compiled_rules` + `reference_sets`; a `migrate-sqlite` CLI verb seeds the db. Editor: `workspace.ts` → **`node:sqlite`** for every entity (`src/lib/server/db.ts`; settings + the Anthropic key stay in `~/.ruleforge-editor.json`, never in the db). On save/delete/test the editor compiles → `compiled_rules` + fires a debounced `POST /admin/refresh` (`src/lib/server/compiled-sync.ts`), so a warm engine reflects edits live — **verified create→serve and delete→404**. Seed from folders: `node scripts/migrate-to-sqlite.mjs "<root>"`.
- **Test runner dock is drag-resizable** (`TestPanel.tsx`) — grip on the top edge, height persisted to `localStorage`, double-click resets.

## Run / verify
- Editor: `npm run dev` → http://localhost:3000.
- Engine: run `../ruleforge` `RuleForge.Api` (port 5050) — or it falls back to the CLI. `/api/test` compiles the editor rule → engine → returns `{ decision, result, trace, timing }`.
- **Claude key:** Settings → AI provider: Claude → paste key (saved to `~/.ruleforge-editor.json`, *outside* the repo), or set `ANTHROPIC_API_KEY`.
- `../ruleforge-sample-workspace/rules/aimeta-demo.json` shows the review-first UI without a key.
- **Typecheck with `npx tsc --noEmit`** (in `ruleforge-editor`). The Next dev server uses SWC and does **not** surface type errors (e.g. a non-exhaustive `switch` over a union) — so always `tsc` after touching shared types/unions. Then runtime/route compile-check by hitting a route (e.g. `GET /rules/<id>`) + watching the dev log.
